# Engineering Principles

이 문서는 이 저장소에서 기능 수정, 버그 수정, 리팩터링을 할 때 항상 우선해야 하는 개발 원칙을 정리한다.

## Core Rule

단위 로직 하나를 고칠 때도 그 로직이 포함된 전체 흐름을 먼저 파악한다.

- 호출하는 쪽과 호출받는 쪽을 함께 본다.
- 입력, 상태 전이, 저장, 집계, 캐시, 화면 반영까지 한 흐름으로 본다.
- 지금 보이는 증상만 막는 임시 수정 대신, 같은 원인이 다른 화면이나 API에서도 다시 터질 수 있는지 확인한다.

## Required Working Style

구현 전에 아래 순서를 기본으로 따른다.

1. 관련 모듈과 데이터 흐름을 먼저 읽는다.
2. 현재 버그나 요구사항이 어느 계층에서 깨지는지 좁힌다.
3. 수정이 퍼지는 영향 범위를 적어도 머릿속으로라도 정리한다.
4. 수정 후에는 연관된 저장, 집계, 캐시, UI 복원 흐름까지 확인한다.

## Design Expectations

가능하면 이미 널리 검증된 소프트웨어 디자인 패턴을 사용한다.

- 복잡한 조건 분기는 helper 또는 strategy-like 구조로 분리한다.
- 읽기 모델과 쓰기 모델이 다르면 aggregate/builder 계층으로 분리한다.
- 화면 로직에서 도메인 규칙을 직접 흩뿌리지 말고 domain helper로 모은다.
- 임시 플래그나 예외 분기보다 명시적인 타입과 모델 확장을 우선한다.
- 캐시 무효화, 집계 재생성, 후처리 같은 cross-cutting logic은 한 곳에 모은다.

## Anti-Patterns To Avoid

- 화면 한 군데만 맞추는 국소 패치
- 데이터 모델 문제를 UI 조건문으로만 덮는 방식
- 연관 API/aggregate/cache를 보지 않고 저장 로직만 수정하는 방식
- 이름만 바꾸고 실제 도메인 모델은 그대로 두는 방식
- 재현 조건 없이 추측으로 고치는 방식

## Definition Of Done

완료라고 말하기 전에 최소한 아래를 만족해야 한다.

- 관련 모듈 흐름을 설명할 수 있다.
- 수정 이유가 도메인 규칙과 맞는다.
- 같은 원인으로 깨질 수 있는 연관 경로를 함께 보완했다.
- 가능한 범위의 테스트 또는 실행 검증을 했다.
