# 5인조 선수교체 사용자 제출형 운영 설계

## 배경

현재 시스템은 사용자 제출 중심으로 이동하고 있다.

- 사용자: 선수등록 제출, 팀편성 제출
- 관리자: 규정 검토 후 승인

하지만 5인조 선수교체는 아직 기존 관리자 운영 화면 중심 사고가 남아 있다. 특히 관리자 스코어보드에서 직접 로스터/전반/후반 라인업을 편집하는 흐름이 남아 있어, 새 운영 철학과 충돌한다.

또한 실제 운영 규칙상 후반 교체는 대회 시작 전에 미리 확정하는 것이 아니라 전반 종료 후 선수 컨디션에 따라 결정될 수 있으므로, 초기 팀편성 시점에 전반/후반을 모두 확정하는 방식은 맞지 않는다.

이 문서는 5인조 선수교체를 `사용자 제출 + 관리자 승인` 흐름으로 재설계하기 위한 기준을 정리한다.

## 목표

- 5인조 팀편성과 후반 교체를 실제 운영 규칙에 맞게 분리한다.
- 사용자가 팀과 후반 교체안을 제출하고, 관리자는 승인만 담당한다.
- 관리자 스코어보드의 직접 편집 의존도를 줄인다.
- 전반/후반 라인업의 승인 이력을 명확하게 남긴다.
- 점수/배정/출력은 승인된 라인업만 기준으로 계산되게 만든다.

## 비목표

- 경기 중 특정 게임부터 교체하는 세밀한 실시간 교체 기능
- 전반 종료 판정을 자동으로 계산하는 복잡한 경기 운영 엔진
- 선수교체와 무관한 일반 종목의 제출/승인 구조 변경

## 운영 규칙

### 1. 팀편성 제출

5인조 팀편성 제출 시 사용자는 다음만 제출한다.

- 로스터 5~7명
- 전반 출전 5명

이 시점에는 후반 출전 5명을 확정하지 않는다.

### 2. 후반 교체 제출

후반 교체는 다음 조건에서만 가능하다.

- 해당 5인조 팀편성이 관리자 승인 완료
- 관리자가 `후반 교체 제출 가능` 상태를 열어둠
- 아직 해당 팀의 후반 교체 제출이 완료되지 않음

사용자는 승인된 로스터 안에서 후반 출전 5명을 한 번 제출할 수 있다.

### 3. 관리자 승인

관리자는 두 단계에서 승인권을 가진다.

- 팀편성 승인
- 후반 교체 승인

후반 교체안이 규정에 맞지 않으면 반려할 수 있고, 반려된 경우 사용자는 재제출할 수 있다.

### 4. 운영 반영

- 전반 점수는 팀편성 승인본의 전반 5명 기준
- 후반 점수는 후반 교체 승인본의 후반 5명 기준
- 승인되지 않은 후반 교체안은 점수/배정/출력물에 반영되지 않음

## 전체 흐름

### 사용자 흐름

1. 선수등록 제출
2. 팀편성 제출
3. 관리자 팀편성 승인
4. 관리자 후반 교체 제출 오픈
5. 사용자 후반 교체 제출
6. 관리자 후반 교체 승인

### 관리자 흐름

1. 팀편성 승인 검토
2. 필요 시 후반 교체 제출 가능 상태 오픈
3. 후반 교체 승인/반려
4. 승인된 라인업 기준으로 운영 확인

## 데이터 설계

## 1. 기존 팀편성 제출

기존 `TeamEntrySubmission`은 유지하되 역할을 축소한다.

- `playerIds`: 전체 로스터
- `firstHalfMemberIds`: 전반 출전 5명
- `secondHalfMemberIds`: 제거 또는 초기엔 사용하지 않음

핵심은 이 문서가 `초기 팀편성 승인`만 담당하게 만드는 것이다.

## 2. 신규 후반 교체 제출

신규 `FivesSubstitutionSubmission` 개념을 추가한다.

권장 필드:

- `id`
- `tournamentId`
- `divisionId`
- `eventId`
- `organizationId`
- `coachUid`
- `teamId` 또는 `teamSubmissionId`
- `rosterIds`
- `firstHalfMemberIds`
- `secondHalfMemberIds`
- `status`
- `createdAt`
- `updatedAt`
- `submittedAt`
- `approvedAt`
- `approvedBy`
- `rejectedAt`
- `rejectedBy`
- `rejectionReason`

이 구조를 통해 초기 팀편성과 후반 교체 이력을 분리한다.

## 3. 이벤트/운영 상태

후반 교체 제출 가능 여부를 이벤트 또는 운영 상태에 저장한다.

권장 필드:

- `fivesSubstitutionWindowOpen: boolean`
- `fivesSubstitutionOpenedAt?: string`
- `fivesSubstitutionOpenedBy?: string`
- `fivesSubstitutionClosedAt?: string`

자동 판정보다 관리자 명시 오픈 방식이 실제 운영에 더 안전하다.

## 화면 설계

## 1. 사용자 팀편성 제출 화면

5인조일 때 보여줄 항목:

- 로스터 5~7명 선택
- 전반 출전 5명 선택

보여주지 않을 항목:

- 후반 출전 5명 선택
- 후반 교체 확정 UI

안내 문구:

- 후반 교체는 전반 종료 후 관리자 오픈 이후 별도로 제출

## 2. 사용자 후반 교체 제출 화면

신규 화면이 필요하다.

표시 내용:

- 승인된 5인조 팀 목록
- 해당 팀 전반 출전 5명
- 벤치 선수
- 후반 출전 5명 선택 UI
- 제출 상태: 미제출 / 승인 대기 / 승인 / 반려

제약:

- 승인된 같은 팀 로스터 안에서만 선택 가능
- 5명 정확히 선택
- 같은 팀당 기본 1회 제출
- 반려 시 재제출 가능

## 3. 사용자 대회관리 허브

기존 액션에 아래를 추가한다.

- `후반 교체 제출`

버튼 상태는 다음처럼 나뉜다.

- 팀편성 미승인: 비활성화
- 관리자 오픈 전: 비활성화
- 제출 가능: 활성화
- 승인 대기/승인 완료: 상태 표시

## 4. 관리자 홈 대시보드

현재 관리자 홈은 요청 현황 대시보드로 사용 중이다.

여기에 다음을 추가한다.

- 승인 대기 후반 교체 요청 수
- 최근 후반 교체 요청 목록
- 바로가기 링크

## 5. 관리자 승인 화면

기존 팀편성 승인과 별도 패널이 필요하다.

표시 내용:

- 팀명
- 로스터
- 전반 출전 5명
- 후반 교체 요청 5명
- 전반 대비 변경 선수
- 제출자/소속/시각

액션:

- 승인
- 반려
- 반려 사유 입력

## 6. 관리자 스코어보드

기본 원칙:

- 직접 선수교체를 하는 운영 화면이 아니라 승인 결과를 반영하는 조회/운영 화면

변경 방향:

- 승인된 후반 교체안이 있으면 해당 후반 라인업 사용
- 없으면 전반 라인업 유지
- 현재 남아 있는 직접 로스터 편집 기능은 기본 운영 경로에서 제거하거나 예외 운영 기능으로 축소

## 서버/API 설계

### 사용자 API

- `GET /api/user/tournaments/[tournamentId]/fives-substitutions`
  - 내 5인조 후반 교체 제출 목록 조회
- `POST /api/user/tournaments/[tournamentId]/fives-substitutions`
  - 후반 교체 제출 생성

검증:

- 승인된 5인조 팀인지
- 요청자가 해당 단체 승인 사용자 맞는지
- 이벤트가 5인조인지
- 제출 창이 열려 있는지
- 이미 승인 대기 중인 제출이 있는지
- 후반 5명 선택이 로스터 안인지

### 관리자 API

- `GET /api/admin/approvals/fives-substitutions`
  - 승인 대기 후반 교체 요청 목록
- `POST /api/admin/approvals/fives-substitutions/[submissionId]`
  - 승인/반려
- `POST /api/admin/tournaments/[tournamentId]/divisions/[divisionId]/events/[eventId]/fives-substitution-window`
  - 제출 창 열기/닫기

## 계산/집계 설계

점수 계산과 출력은 `전반/후반 시점별 활성 라인업` 기준으로 돌아야 한다.

현재 코드에는 다음 공통 로직이 이미 존재한다.

- `src/lib/fives-lineup.ts`
- `src/lib/team-lineup.ts`
- `src/lib/scoring.ts`

이 공통 로직은 유지하되, 후반 라인업의 소스만 바꾼다.

- 기존: 팀 문서의 `secondHalfMemberIds`
- 변경 후: 승인된 후반 교체안이 있으면 그 값 우선

### 사용자

- `src/components/user/TeamSubmissionManager.tsx`
  - 5인조 제출을 `로스터 + 전반 5명`으로 축소
- 신규 `후반 교체 제출` 페이지/컴포넌트
- 사용자 대회관리 허브에 `후반 교체 제출` 액션 추가

### 서버

- 신규 후반 교체 제출 컬렉션/타입 정의
- 사용자 제출 API
- 관리자 승인 API
- 관리자 제출창 오픈 API

### 관리자

- 관리자 홈 대시보드에 후반 교체 대기 현황 추가
- 관리자 승인 패널 추가
- 스코어보드의 직접 교체 UI 축소 또는 제거

### 집계/반영

- 팀 승인 프로젝션은 전반 기준으로 생성
- 후반 교체 승인 프로젝션이 후반 라인업을 별도 반영
- 스코어/출력은 승인본 참조

## 단계별 구현 순서

1. 사용자 팀편성 제출에서 후반 선택 제거
2. 후반 교체 제출 타입/컬렉션 추가
3. 사용자 후반 교체 제출 화면/API 추가
4. 관리자 오픈/승인 API 추가
5. 관리자 대시보드/승인 패널 추가
6. 스코어보드와 출력물이 승인된 후반 라인업을 읽도록 변경
7. 기존 관리자 직접 교체 UI 정리

## 테스트 전략

- 팀편성 제출 시 5인조는 전반만 저장되는지
- 후반 교체는 오픈 전 제출 불가인지
- 승인된 로스터 밖의 선수는 제출 불가인지
- 후반 교체는 5명 정확히 선택해야 하는지
- 승인 전에는 후반 점수 반영이 되지 않는지
- 승인 후에는 후반 점수/배정/출력에 반영되는지
- 반려 후 재제출 가능한지

## 위험 요소

- 현재 관리자 스코어보드에 직접 수정 로직이 깊게 남아 있어 충돌 가능성 있음
- 기존 `secondHalfMemberIds` 사용처가 많아 점진적 치환 필요
- 팀 승인과 후반 교체 승인 간 참조 무결성을 잘 잡아야 함

## 권장 결론

5인조 선수교체는 더 이상 관리자 수동 편집 기능이 아니라, 사용자 운영 제출의 일부로 본다.

초기 팀편성은 `로스터 + 전반`, 후반은 `전반 종료 후 별도 제출`, 관리자는 `오픈 + 승인`만 담당하게 역할을 재정렬하는 것이 현재 시스템 방향과 가장 잘 맞다.
